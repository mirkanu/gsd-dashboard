<task type="auto">
  <name>Task 1: Verify tunnel prerequisites and local server health</name>
  <files></files>
  <action>
Verify the local environment is ready before configuring Railway:

1. Check cloudflared is installed and can reach the tunnel credentials:
   - Run: cloudflared tunnel list
   - This lists named tunnels. If it fails with "login required", the user must run tunnel-setup.sh first.

2. Check that the local server is running on port 4820:
   - Run: curl -s http://localhost:4820/api/health
   - Expected: {"status":"ok","timestamp":"..."}

3. Check that CLOUDFLARE_TUNNEL_NAME is set in .env (or environment):
   - Run: grep CLOUDFLARE_TUNNEL_NAME .env 2>/dev/null || echo "NOT SET"

4. Check if GSD_DATA_URL is already configured in .env (it should NOT be — it belongs on Railway only):
   - Run: grep GSD_DATA_URL .env 2>/dev/null || echo "not in .env (correct)"

5. Start the named tunnel in the background for testing:
   - Run: CLOUDFLARE_TUNNEL_NAME=$(grep CLOUDFLARE_TUNNEL_NAME .env | cut -d= -f2) nohup sh scripts/tunnel.sh > /tmp/gsd-tunnel.log 2>&1 &
   - Wait 10 seconds for it to establish
   - Run: grep -i "tunnel" /tmp/gsd-tunnel.log | head -5

6. Verify tunnel is reachable:
   - Run: curl -s https://YOUR_HOSTNAME/api/health (replace YOUR_HOSTNAME with the configured tunnel hostname)
   - Expected: {"status":"ok",...}

If any step fails, stop and report the exact failure. Do NOT proceed to Railway configuration until the tunnel is confirmed reachable.
  </action>
  <verify>
    <automated>curl -s http://localhost:4820/api/health | grep -q '"status":"ok"' && echo "local server OK"</automated>
  </verify>
  <done>Local server responds on port 4820, cloudflared is installed, CLOUDFLARE_TUNNEL_NAME is set, and tunnel is confirmed reachable via its permanent HTTPS URL</done>

<task type="auto">
  <name>Task 2: Set GSD_DATA_URL on Railway and deploy</name>
  <files></files>
  <action>
Configure Railway with the permanent tunnel URL and redeploy.

1. Set the Railway environment variable using the CLI:
   ```
   cd /data/home/gsddashboard
   railway variables --set "GSD_DATA_URL=https://YOUR_HOSTNAME"
   ```
   Replace YOUR_HOSTNAME with the actual tunnel hostname from .env or the cloudflared tunnel list output.

2. Deploy to Railway with the new env var:
   ```
   railway up --detach
   ```

3. Wait for deployment to complete. Check status:
   ```
   railway status
   ```
   Wait until the deployment shows as "ACTIVE" or "SUCCESS".

4. Verify the Railway deployment picked up the new GSD_DATA_URL by hitting the ws-base endpoint:
   ```
   curl -s https://YOUR_RAILWAY_URL/api/gsd/ws-base
   ```
   Expected response: {"wsBase":"wss://YOUR_HOSTNAME"} (not {"wsBase":null})

   The Railway URL can be found from: railway open (prints URL) or existing deployment URLs.

5. Verify Railway can reach the local server through the tunnel:
   ```
   curl -s https://YOUR_RAILWAY_URL/api/gsd/projects
   ```
   Expected: {"projects":[...],"rateLimit":false} — projects from the local gsd-projects.json, not an error.

Note: If railway CLI is not available, provide the Railway dashboard URL and exact steps to set the env var manually via the web UI.
  </action>
  <verify>
    <automated>railway variables 2>/dev/null | grep -q "GSD_DATA_URL" && echo "GSD_DATA_URL set on Railway"</automated>
  </verify>
  <done>Railway has GSD_DATA_URL set to the permanent tunnel hostname, deployment is active, and /api/gsd/ws-base returns a non-null wsBase pointing to the tunnel</done>
</task>
